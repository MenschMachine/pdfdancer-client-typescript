The pdfdancer-api project just integrated a new capability.
Your task is to expose that capability to this client SDK.

Read the capability files below carefully. They describe what was added on the API side.

Your job:
1. Understand what the new API capability does
2. Design the new SDK feature. The API should be as simple and user-friendly as possible. Following existing patterns strictly.
3. Create e2e tests for the new API. Following the existing patterns. All e2e tests must have deep and precise assertions using the PDFAssertions class. Add new methods to the PDFAssertions class if needed.
4. Failing tests may expose bugs. That's their job. In case you encounter a server-side bug, be sure there is a failing test for it and stop. Write a detailed bug report as a markdown file to be picked up by the pdfdancer-api team.

## Starting the API Server

Use the docker image from the upstream pdfdancer-api PR listed in the "Upstream PRs" section below.
The image tag follows the pattern `ghcr.io/menschmachine/pdfdancer-api:pr-{NUMBER}`.

```shell
docker pull ghcr.io/menschmachine/pdfdancer-api:pr-{NUMBER}
docker run \
    -e PDFDANCER_API_KEY_ENCRYPTION_SECRET=`openssl rand -hex 16` \
    -e FONTS_DIR=/tmp/fonts \
    -e METRICS_ENABLED=false \
    -e SWAGGER_ENABLED=true \
    -v /tmp/fonts:/home/app/fonts \
    --rm \
    -p 8080:8080 \
    ghcr.io/menschmachine/pdfdancer-api:pr-{NUMBER}
```

The port maybe occupied by another process. In that case, try another port.
API docs are available at http://localhost:8080/swagger-ui

# Clear Clipping Capability

Most of the PDF model classes including container classes like PDFParagraph/PDFTextLine/PDFPathGroup now implement ClippingDetachable with the method clearClipping().
This removes any clipping path which was active for this element.

This is useful in case clients want to move an element but the new position is hidden by the clipping path. clearing it makes the element visible again.

The new backend is version 1.8.6-rc2, available in the local m2 repository
